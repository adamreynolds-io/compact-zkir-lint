# RT-003: transient_hash with guarded inputs

**Severity:** info | **Fixable:** manual | **Since:** 0.1.0

## What it detects

`transient_hash` instruction where one or more input variables are from a guarded region (conditional branch).

## Why it matters

JS and ZKIR encode hash inputs differently:

- **JS** (`compact-runtime`): calls `type.toValue()`, which for `CompactTypeBytes` strips trailing `0x00` bytes
- **ZKIR** (`ir_vm.rs`): receives raw field elements without the trailing-zero truncation

When guarded inputs default to 0, the JS serialization (empty array after stripping) differs from ZKIR's serialization (field element 0). This produces different hash outputs between the two execution paths.

Severity is **info** because `transient_hash` is used for ephemeral commitments that don't persist on-chain. The divergence may not cause proof failure depending on how the hash is used downstream.

## Examples

### Transient hash inside branch (may diverge)

```compact
pragma language_version >= 0.22.0;

export circuit hash_if_needed(
  data: Bytes<32>,
  should_hash: Boolean
): Field {
  if (should_hash) {
    return transient_hash(data);  // JS strips trailing zeros; ZKIR uses raw fields
  } else {
    return 0 as Field;
  }
}
```

### Fixed: hash before the branch

```compact
pragma language_version >= 0.22.0;

export circuit hash_if_needed(
  data: Bytes<32>,
  should_hash: Boolean
): Field {
  const hash = transient_hash(data);
  return should_hash ? hash : 0 as Field;
}
```

## Technical details

The divergence is in `CompactTypeBytes.toValue()` (`compact-types.ts:379`), which truncates trailing `0x00` bytes. ZKIR receives the full field element representation. When the hash inputs are all zeros (dead branch), JS produces an empty byte array while ZKIR produces the field element 0 — these hash to different values.
