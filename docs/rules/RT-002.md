# RT-002: less_than with guarded operands

**Severity:** info | **Fixable:** manual | **Since:** 0.1.0

## What it detects

`less_than` instruction where one or both operands are from a guarded region (conditional branch), and the operands are not guaranteed to be zero when the guard is false.

## Why it matters

ZKIR `less_than` extracts N bits from each operand before comparing the truncated values. In a dead branch, guarded operands default to 0. The comparison result (`0 < X` or `X < 0`) may be incorrect, but it typically feeds into a `cond_select` that discards it anyway.

This is severity **info** because the incorrect comparison result is almost always discarded by the branch merge. However, the bit extraction itself involves internal constraints that fire on the dead-branch values.

## Examples

### Comparison inside branch (flagged)

```compact
pragma language_version >= 0.22.0;

ledger result: Uint<64>;

export circuit min_in_branch(
  a: Uint<64>,
  b: Uint<64>,
  use_min: Boolean
): [] {
  if (use_min) {
    // less_than extracts bits from a and b unconditionally
    result = a < b ? a : b;
  } else {
    result = a;
  }
  return [];
}
```

### Fixed: compare before the branch

```compact
pragma language_version >= 0.22.0;

ledger result: Uint<64>;

export circuit min_in_branch(
  a: Uint<64>,
  b: Uint<64>,
  use_min: Boolean
): [] {
  const min_val = a < b ? a : b;
  result = use_min ? min_val : a;
  return [];
}
```
